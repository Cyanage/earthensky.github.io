Optimization :
  • False Sharing: When a (global) variable is accessed (shared) by more than one thread. If one thread writes to the variable, each cpu core's cache has to update the variable's data from the ram (slow).
    Fix: Always operate on local variables inside a thread. Once the processing is done, use those values to update a global state.
    Quip: --Why do you think people say don't use global variables?--
    See: using the cpu chache.

  • Cached Matrixes: If you create a 2-D matrix, it’s row major traversal will be faster than it’s column major traversal. A row is stored in consecutive memory locations and is thus fetched in a cache line.
    See: using the cpu chache.

  • CPU Guesses: A cache is also smart enough to prefetch the previous or next cache line required, depending upon the instruction. So, the more predictable is your access pattern the better will be the performance.
    See: using the cpu chache.

  • Good Cache Knowledge:
    - Maintain locality of data. Repeated references to a local variable are good because it'll be in cache.
    - Accessing memory in a sequential manner means utilising the already loaded words in cache line and thus, less time accessing the ram.
    - Avoid unnecessary memory allocations. Memory is costly and it’s access is not as fast as it seems. Don’t unnecessarily copy objects from one place to another (unless you are dealing with immutable objects in functional programming).
    See: using the cpu chache.
